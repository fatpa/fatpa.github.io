Vibe Coding

Vibe Coding 真的可以让产品快速 MVP

工作这么多年记忆最深刻的是，每次启动一个新项目总会需要投入大量的人力去筹划预研，输出产品文档，开无数个会评审需求，输出技术方案，再到最后的开发测试上线。耗时久不说，随时随地就会在中间被关闭，半途而废的项目数不胜数。而且作为一名只会写后端的工程师，有时候也很苦恼于一些小创意无法快速实现。但是自从 Vibe Coding 流行起来后，我发现这个事情似乎已经没有以前那么困难了，甚至是说，真的是到了「人人都是产品经理」的时代了。 作为一个之前只用 copilot 代码提示的研发，最近闲着用 ClaudeCode Cli 对一个想法进行了简单的尝试，没想到越用越上瘾，从一开始的只是想做个 demo 体验下整个自动编码过程，到最后仅仅花了四个早上的时间，就把一个付费 App 全流程都实现跑通了，并基于 Cloudflare Pages 和 Zeabur 直接部署上线。很突然但也很兴奋，原来现在对想法进行 MVP 已经变得这么简单了。 虽然说是叫 Vibe Coding，但实际体验下来这几个点还是绕不开的，如果没有，可能整个过程会更漫长： 一丢丢的研发运维基础，不至于让自己连个本地环境都跑不起来，出个什么问题都需要去问 AI 排查； 配置好自己的 CLAUDE.md，不要安装好 cli 就开始吭哧吭哧干活，如果有时间最好把 agents 和 skills 都慢慢补齐优化； 一定一定要用好 Plan Mode，一上来不要就让 AI 去干活，而是想跟他沟通，交流你的想法，让它给输出个完整的技术方案，你 review 方案再提交改进意见，always human in the loop，像工作中的开会评审一样，最终大家都建立好共识，这时候再开始让它去编码； 要限制好你的项目规范，规矩要先立好，该用什么框架，该用什么目录结构去存放代码，什么模块要抽象统一，这些都是要在 CLAUDE.md 里配置好的，不要让它在整个编码过程放飞自己，要不然到最后你要自己动手调整代码都无从下手； 做好架构师的角色，把自己需要用到的 SaaS 服务或者框架组件都提前了解好，清楚知道每个组件的功能用途，在具体 vibe 过程告诉 AI 这些细节； 不要一口气让它去干一堆活，自己提前把功能想法抽象好，每次只让它专注地做一件事情，就把它当作一个中级研发看待，专心干好每个需求； 每完成一个需求时，都要亲身去体验每个功能细节，不能太过于相信 AI。 相信如果有这么些前置了解，Vibe Coding 起来会更加顺手的。 最后，不得不夸夸现在的 SaaS 服务真的又细分又方便，皆是赛博活菩萨。这次一人研发的项目，就是基于 Supabase + Cloudflare Pages + Zeabur + DeepSeek API 实现了前后端服务部署和数据库存储的能力，也引入了 DeepSeek LLM 进行了文本分析归类，全程没有敲过一行指令，打包构建部署都也完全模板化。所以更加笃定大家以后要验证一个产品的 MVP 会更加的简易方便了，短短几天时间就可以上线开始测试验证，再持续调整，根本不可能给老板们时间去犹豫。 ...
